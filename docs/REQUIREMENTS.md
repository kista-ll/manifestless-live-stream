# 要求仕様

## 1. 背景

HLS/DASHのようなマニフェストを使わず、セグメント型メディアをPush配信してブラウザ再生できるかを検証する。

## 2. システム境界

対象:

- SRTによるエンコーダー入力の受信
- FFmpegによるMPEG-TSのdemuxとfMP4セグメント生成
- 生成済みセグメントの検出とメモリ保持
- WebTransportによる視聴者へのPush
- MSEによるブラウザ再生
- 接続、再接続、遅延追従、配信終了

対象外:

- 商用運用
- 認証・認可
- DRM
- CDN
- 録画・VOD
- 追いかけ再生UI
- ABR
- 複数配信
- 視聴分析
- Safari対応保証
- モバイル対応保証

## 3. 機能要求

### FR-001 SRT配信入力

配信サーバーはSRT Listenerとして接続を待ち受け、エンコーダーから以下の形式を受信できること。

- 接続モード: Encoder = Caller / Server = Listener
- トランスポート: SRT
- コンテナ: MPEG-TS
- 映像: H.264/AVC
- 音声: AAC-LC
- 待受ポート: UDP 9000

初期受入試験では、FFmpegのテスト映像をSRT Callerとして送信する。

### FR-002 CMAF/fMP4生成

SRT ingest用FFmpegは、受信したH.264/AACを以下へ変換すること。

- `init.mp4`
- 1秒単位の`segment-{sequence}.m4s`
- 各メディアセグメントは独立再生可能なGOP境界で開始する

入力が指定されたコーデック・GOP条件を満たす場合はstream copyを使用する。条件を満たさない場合は、実験設定に合わせて再エンコードする。

### FR-003 セグメント検出

サーバは出力ディレクトリを監視し、新しいセグメントをsequence順に登録すること。

### FR-004 リングバッファ

サーバは直近30セグメントをメモリ上で保持すること。31個目の追加時に最古を破棄する。

### FR-005 視聴開始

視聴者接続時、以下を順に送ること。

1. `stream_init`制御メッセージ
2. init segment
3. 接続時点の最新から2セグメント前を起点としたmedia segment
4. 新規生成されるmedia segment

保持セグメントが2個未満なら最古の保持セグメントから開始する。

### FR-006 ライブPush

サーバは新しいmedia segmentを接続中の全視聴者へPushすること。視聴者からのセグメント一覧ポーリングを必要としないこと。

### FR-007 最大視聴者数

同時接続数は10とする。11人目には`capacity_exceeded`を通知し、セッションを終了する。

### FR-008 プレイヤー

プレイヤーは以下を行うこと。

- `MediaSource.isTypeSupported()`でMIME typeを確認
- init segmentを最初にappend
- media segmentをsequence順にappend
- SourceBuffer更新中はキューへ保持
- 直近5秒より古いバッファを削除
- 遅延が5秒を超えた場合、live edgeの2.5秒前へ移動
- 遅延が3秒を超え5秒以下の場合、再生速度を1.05にする
- 遅延が3秒以下の場合、再生速度を1.0に戻す

### FR-009 再接続

接続切断後、プレイヤーは1秒、2秒、4秒、8秒、最大10秒の間隔で再接続すること。再接続時は新しいMediaSourceを作り直して最新位置から再生する。

### FR-010 配信終了

セグメンター終了をサーバが検知または管理APIで終了を指示された場合、`stream_ended`を送ること。プレイヤーはキューを消化後に`endOfStream()`を呼ぶこと。

### FR-011 状態表示

視聴画面に以下を表示すること。

- 接続状態
- 再生状態
- 最新受信sequence
- 推定ライブ遅延
- バッファ秒数
- エラー

## 4. 非機能要求

### NFR-001 同時視聴

ローカル環境または同一LAN上で10視聴者が同時に接続できること。

### NFR-002 遅延

安定したローカルネットワークで、再生開始後30秒時点のライブ遅延が2〜5秒に収まること。

### NFR-003 起動時間

配信開始後10秒以内に新規視聴者が再生開始できること。

### NFR-004 欠落耐性

1つのmedia segmentが欠落した場合、プレイヤーは停止し続けず、次の独立セグメントから復旧すること。

### NFR-005 可観測性

サーバログに以下を含めること。

- session ID
- viewer接続・切断
- viewer数
- segment登録
- segment送信
- sequence欠落
- エラー

### NFR-006 ローカル再現性

Docker ComposeとMakefileを使い、READMEの手順だけで起動できること。

### NFR-007 SRT再接続

SRT Callerが一時切断後に再接続した場合、サーバーは新しいingest sessionを受け入れられること。

### NFR-008 入力検証

映像または音声track不足、未対応codec、異常なGOPを検出した場合、原因をログへ出し、stream stateをERRORまたはINTERRUPTEDへ遷移させること。

## 5. 受入条件

### AC-001 基本再生

FFmpegテスト映像を開始し、Chromeで視聴ページを開くと10秒以内に映像・音声が再生される。

### AC-002 途中参加

配信開始から60秒後に視聴ページを開き、過去全体を取得せず10秒以内に最新付近から再生される。

### AC-003 10視聴

10個のブラウザコンテキストが同時にPLAYINGになる。

### AC-004 11人目拒否

11人目が`capacity_exceeded`となり、既存10人の再生が継続する。

### AC-005 再接続

サーバ接続を5秒遮断して戻すと、プレイヤーが自動再接続し30秒以内にPLAYINGへ戻る。

### AC-006 遅延追従

プレイヤーを10秒間一時停止して再開すると、30秒以内にライブ遅延5秒以下へ戻る。

### AC-007 配信終了

配信終了後、視聴画面がENDEDになり、無限再接続しない。

### AC-008 マニフェスト不使用

実行中のHTTP/HTTP3通信に`.m3u8`、`.mpd`、またはセグメント一覧レスポンスが存在しない。

### AC-009 SRT ingest

SRT CallerからH.264/AACをMPEG-TSで送信すると、配信サーバーが10秒以内にLIVE状態となり、`init.mp4`とmedia segmentの生成を開始する。

### AC-010 SRT再接続

SRT Callerを停止し、10秒後に再開すると、配信サーバーが新しい接続を受け入れ、新しいinit segmentを生成し、viewerが30秒以内にPLAYINGへ復帰する。

### AC-011 不正入力

映像track不足、音声track不足、未対応codecのいずれかを受信した場合、stream stateがERRORまたはINTERRUPTEDになり、原因がログへ記録される。
